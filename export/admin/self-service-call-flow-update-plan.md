# แผนปรับปรุง Self Service Call Flow HTML

## เป้าหมาย

ปรับปรุงไฟล์ `gate-flow` เดิมให้ข้อมูล IP ถูกต้องและลดความเสี่ยงการ route เคสผิด โดยคง layout, flow, section, script และดีไซน์เดิมให้มากที่สุด

## หลักการแก้ไข

- คงโครง HTML เดิม 100%
- ไม่เปลี่ยน flow diagram เดิม
- ไม่รื้อ layout หรือ design
- ไม่เพิ่ม section ใหม่ถ้าไม่จำเป็น
- แก้เฉพาะข้อมูล IP และข้อความกำกับเล็กน้อย
- ใช้ `knowledge/ip-ranges.yaml` เป็น source of truth สำหรับ IP ranges

## จุดที่ควรปรับข้อความเล็กน้อย

### 1. เพิ่มความชัดเจนเรื่อง Portal/IP ขัดกัน

ตำแหน่งแนะนำ:

ใน note เดิมที่เขียนประมาณว่า:

```text
กฎสำคัญ: Email ไม่ใช่ตัวตัดสินว่าลูกค้าอยู่ฝั่ง Kory หรือ Gate ต้องถาม Portal ก่อนเสมอ และถ้าเป็นเคส Instance / VM / Network ให้ใช้ Public IP ช่วยยืนยันฝั่งบริการ
```

ปรับเป็น:

```text
กฎสำคัญ: Email ไม่ใช่ตัวตัดสินว่าลูกค้าอยู่ฝั่ง Kory หรือ Gate ต้องถาม Portal ก่อนเสมอ และถ้าเป็นเคส Instance / VM / Network ให้ใช้ Public IP ช่วยยืนยันฝั่งบริการ หาก Portal และ Public IP ขัดกัน ให้ส่งเข้า Email / Verify เพื่อขอข้อมูลเพิ่มและตรวจสอบก่อนเปิดเคส
```

เหตุผล:

- กันเคสที่ลูกค้าบอก Portal อย่างหนึ่ง แต่ IP อยู่คนละฝั่ง
- ไม่ต้องเพิ่ม flow ใหม่
- คงโครงเดิม แต่ลดการ route ผิด

### 2. ปรับ Email-only ให้เป็น last resort ชัดเจน

ตำแหน่งเดิม:

```text
ถ้ามีเพียง email และสุดวิสัยจริง ๆ ให้เปิดเคสไปที่ Kory ก่อน เพื่อให้ OP OLS ตรวจสอบต่อ
```

ข้อความนี้ดีอยู่แล้ว แต่ถ้าต้องการให้เข้มขึ้น แนะนำเป็น:

```text
ถ้ามีเพียง email และสุดวิสัยจริง ๆ หลังพยายามขอ Portal/Public IP แล้ว ให้เปิดเคสไปที่ Kory ก่อน โดยระบุว่า platform ยังไม่ยืนยัน เพื่อให้ OP OLS ตรวจสอบต่อ
```

เหตุผล:

- กันทีมใช้ Email-only เป็น default
- ยังรักษา rule เดิมว่า last resort เปิด Kory ก่อน

### 3. เพิ่ม source ของ IP แบบสั้น

ตำแหน่งแนะนำ:

ใต้ section IP cards หรือ footer

ข้อความแนะนำ:

```text
หมายเหตุ: IP ranges อ้างอิงจาก knowledge/ip-ranges.yaml
```

หรือถ้าต้องการใส่วันที่:

```text
หมายเหตุ: IP ranges อ้างอิงจาก knowledge/ip-ranges.yaml อัปเดตล่าสุด 2026-06-24
```

เหตุผล:

- คนอ่านรู้ว่า IP มาจาก source ไหน
- ลดปัญหาใช้ไฟล์เก่าในอนาคต

## สิ่งที่ไม่ควรเปลี่ยน

- ไม่ควรเปลี่ยน layout หลัก
- ไม่ควรเปลี่ยนชื่อ section
- ไม่ควรเปลี่ยน Main Decision flow
- ไม่ควรเปลี่ยน lane Kory / Gate / Email / Commercial
- ไม่ควรเปลี่ยน Customer Scripts ถ้าไม่จำเป็น
- ไม่ควรเพิ่ม Quick Decision Table ใหม่ หากต้องการคงหน้าตาเดิม
- ไม่ควรเปลี่ยน wording เยอะ เพราะอาจทำให้ทีมสับสนกับ flow เดิม

## Checklist หลังแก้

ตรวจสอบว่า:

- ไม่มี `203.150.82.0/24` เหลืออยู่
- `203.154.82.0/24` อยู่ฝั่ง Gate
- `203.154.91.0/24` อยู่ฝั่ง Gate
- `203.154.91.0/24` ไม่อยู่ฝั่ง Commercial
- ข้อความ Email-only ยังระบุว่าเป็นกรณีสุดวิสัย
- มีคำแนะนำกรณี Portal/IP ขัดกัน
- Layout เดิมไม่เปลี่ยน
- เปิดบน desktop/mobile แล้วยังอ่านได้เหมือนเดิม

## สรุปแนวทางที่แนะนำ

ให้แก้แบบ minimal change:

1. แก้ IP ผิดฝั่ง
2. เพิ่มข้อความสั้น ๆ เรื่อง Portal/IP ขัดกันใน note เดิม
3. เพิ่มความชัดเจนว่า Email-only เป็น last resort
4. เพิ่ม source IP ranges ท้าย section
5. ไม่แตะ flow/layout/script อื่น ๆ
